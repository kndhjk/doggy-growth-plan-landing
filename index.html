项目亮点：不只是 proposal，而是做出来了

下面这些点都直接能对应到 repo 里的前端页面、API 处理器和部署说明，适合拿来打 rubric 里的 Functionality、Technical Ambition 和 Testing & Deployment。

Digital Pet Core

宠物创建、头像成长阶段、4 状态衰减、照片上传、档案编辑，不是静态展示，而是持续演化的宠物状态机。

  • CreatePetModal + PetContext + status decay
  • 本地 fallback 让 demo 在后端缺失时仍可跑

AI + Context Injection

AI 顾问不是纯聊天壳。它把 breed、年龄和当前 4 状态注进 prompt，再通过 Gemini 或本地 fallback 给出建议。

  • Gemini API handler + fallback responses
  • 中文优先，更适合真实用户习惯

Community + Live Interaction

社区页包含发帖、点赞、评论，配合 Firestore 实时同步；本地模式下也保留 localStorage 体验，适合演示。

  • 狗视角日记 + 情绪反应 + 成就墙
  • Matchmaking 与 chat 可作为社交扩展点

Map + real-world use

地图不是装饰页。它结合定位、Places 查询和 check-in，让散步、宠物医院、宠物店这些真实场景进入产品。

  • Google Places 代理 + nearby search
  • 有 demo 数据，也能接真实定位

Beyond Proposal Modules

Marketplace、Adopt Centre、Inventory、Leaderboard、Rewards、Training、Health Records，把产品从“宠物仪表盘”扩成多 tab 平台。

  • 这部分最适合打 Technical Ambition
  • 也说明团队没有只停在最低交付线

Design + i18n

粉色设计系统、卡片化布局、framer-motion 动画、多语言扩展和本地持久化，让演示看起来像真正产品而不是作业截图。

  • 视觉统一感强，适合现场 demo
  • 多语言也有现实可用性

怎么对准评分标准

这部分不是照抄 rubric，而是帮你们现场更像在“主动交作业”。

Functionality & User Stories

不要只说“我们有很多 tab”。要说清楚完整链路：创建宠物 → 状态追踪 → AI 建议 → 地图 check-in → 社区互动 → 扩展服务。

Project Management & Collaboration

你给的 division of work 本身就很强。演讲时直接把“谁负责哪条功能链路”说清楚，会比泛泛说 teamwork 更有说服力。

Technical Ambition & Research

你们最能打的点：Gemini、Google Places、Firebase 实时同步、情绪反馈联动、beyond proposal 的 v3 模块扩展。

Testing & Deployment

虽然不是 full production stack，但 repo 里已经有测试服务器部署记录、依赖说明和稳定的前端静态部署路径，这比“只在本地跑通”更强。

Code Quality & Architecture

讲模块边界：页面、context、components、API handlers、fallback 设计。重点不是夸代码，而是说明你们知道为什么这样拆。

最后 20 秒收尾模板

我们不只是做了一个看起来可爱的宠物页面，而是把实时状态、AI 建议、社区互动、地图场景和扩展服务整合成了一套完整的狗狗成长平台。
