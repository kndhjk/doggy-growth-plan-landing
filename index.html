系统分为三层

整个 GG Bond 平台分成三层。每一层只做自己的事，互不干扰。

Layer 1 — 前端
用户看到和操作的部分
React · React Router · Framer Motion · Firebase
🎨
Layer 2 — 后端
处理业务逻辑
Node.js · Express · RESTful API · mysql2
⚙️
Layer 3 — 数据库
保存所有数据
MySQL 8.0 · 14 tables · 7 functional modules
🗄️
可以这样讲： 很好记：前端管界面，后端管逻辑，数据库管存储。三层各做各的事，所以改其中一层不会影响另外两层。

接口设计：前后端怎么对话

我们用 RESTful 规范来设计接口。每个操作对应一个独立的 URL 地址。

GET /pets

查询某只宠物的信息

PUT /pets/:id

更新宠物状态

POST /market

发布商品到市场

POST /chat

发送聊天消息

GET /achievements

获取成就列表

POST /rewards/claim

领取每日奖励

好处是：前端只管调接口，不用知道数据存在哪、怎么存的。前后端完全分开，改前端不影响后端，反过来也一样。

👤 Xiang Zhang · Part 2 · Slide 2/3

数据库操作：安全和稳定

参数化查询

所有查询都用参数拼接，防止 SQL 注入攻击。

db.query(
'SELECT * FROM pets
WHERE user_id = ?',
[userId])
// ? = 用户ID，不会被注入

连接池

MySQL 驱动自己管理连接池，多人同时访问也不会崩。

mysql2.createPool({
 host: 'localhost',
 database: 'ggbond',
 waitConnections: true,
 queueLimit: 0
})

简单说：mysql2 库让我们直接写 SQL 查询，但通过参数化和连接池保证了安全性和并发稳定性。

模板表设计：一份技能，所有狗通用

这里用了模板表加记录表的设计思路。以训练系统为例：

📋 模板表
training_skills
Sit
Shake Hand
Lie Down
Stay
存的是技能定义，所有狗共用一套
📝 记录表
pet_training_progress
Cookie · Sit · done
Cookie · Stay · 60%
Max · Lie Down · done
存每只狗的进度，各不相同
🎯 好处一：数据不重复
技能定义只存一份，所有狗都引用它。
🧩 好处二：好扩展
加新技能只改模板表，不用动记录表。

核心模块：宠物照护、训练、成就

🐾 宠物照护系统

  • pet_activities — 日常活动记录
  • health_records — 健康和医疗历史

喂食、散步、玩耍、看兽医等都记录在这里。

🎓 训练系统

  • training_skills — 技能模板
  • pet_training_progress — 每只狗的进度

模板表存技能定义，记录表存每只狗学到哪了。

🏆 成就系统

  • achievements — 成就定义
  • pet_achievements — 解锁记录

First Meal、First Walk、Brave Pup 等都从这里来。

共同点：都用了模板表加记录表结构。这种设计让数据可以复用，系统也更容易扩展。

👤 DB · Part 3 · Slide 3/3

其他模块和整体关系

🎁 每日奖励

  • daily_rewards — 奖励模板，第1天金币、第2天食物…
  • user_reward_claims — 每个用户的领取记录

🛒 市场 + 背包

  • marketplace_listings — 用户发布的商品
  • inventory — 用户拥有的物品

💬 聊天系统

  • conversations — 会话容器
  • messages — 具体消息

M:M 用户参与会话，1:M 会话包含消息。

📊 全貌总结

  • 总计 14 张表，7 个功能模块
  • users 和 pets 是核心，其他模块向外扩展
  • 模板表+记录表用于训练、成就、奖励三个系统
  • 模块之间低耦合，便于单独开发和扩展

这就是 GG Bond 的完整数据库设计。14 张表围绕 users 和 pets 两个核心，7 个模块各司其职，模块之间低耦合、易扩展。
